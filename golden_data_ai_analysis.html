BUG-4: Cardiac Arrest Alarm Fires but Cycles ON/OFF (Partially Fixed) P1 — PATIENT SAFETY
Symptom
The invalid-pair alarm check was added (source fix confirmed) — "Cardiac Alarm3 ON (invalid pairs)!" now fires in scenarios 23 (6 cycles) and 24 (1 cycle). However, the alarm is immediately cancelled by the alarm-OFF logic within seconds. Scenarios 21 (VF) and 22 (VT→VF) still produce zero alarms.
Affected Scenarios
21_vf_from_sinus (0 alarms), 22_vt_to_vf (0 alarms), 23_asystole (6 alarm ON/OFF cycles, cardiac_arrest=2 for 1 of 60 records), 24_pea (1 alarm ON/OFF cycle, cardiac_arrest=2 for 1 of 60 records)
Root Cause (Remaining Issue)
The alarm-OFF logic at corsano_cardiac_arrest.c:287 requires only 4 consecutive above-threshold peaks to cancel the alarm. The inter-peak timing validation (lines 302-330) that would verify realistic cardiac intervals is commented out. During arrest, noise produces occasional spurious peaks that satisfy the unvalidated condition, causing rapid alarm ON/OFF cycling.
Evidence
Scenario 23 (asystole) log shows 6 complete alarm cycles: 
// corsano_cardiac_arrest.c:349-359 -- FIX APPLIED
} else {
    below_threshold_duration += time_diff;
    // BUG-4 FIX: alarm check in invalid-pair branch
    if (below_threshold_duration > min_duration && !alarm_on) {
        alarm_on = true;
        corsano_cardiac_set_state(CARDIAC_ARREST);
        LOG_TO_FLASH("Cardiac Alarm3 ON (invalid pairs)!");
    }
}
// But alarm-OFF at line 287 cancels within seconds:
// consecutive_above_threshold >= 4  →  alarm OFF (no timing validation)
Scenario 23: Alarm3 ON fires 5 times, Alarm2 ON fires 1 time, each cancelled within ~20-50 log lines. cardiac_arrest=2 for only 1 of 60 activity records.
Fix (Remaining)
Un-comment the 2-second inter-peak validation at lines 302-330. Require sustained normal rhythm (15+ consecutive above-threshold peaks within physiological timing) before turning alarm OFF. Add hysteresis: alarm-ON requires 10s, alarm-OFF should require 30s of normal rhythm.

Symptom (resolved)

Root Cause (was)
Local variable float last_peak_above_threshold = 0.0; at corsano_cardiac_arrest.c:233 shadowed the global at line 143. Every function call started with the local at 0.0, making the gap condition always false.
Resolution

Scenario 23 fires "Cardiac Alarm2 ON!" (amplitude-based, valid pair) — also requires the global to persist.
NEW: Alarm-OFF Logic Too Permissive + Counter Reset by Spurious Peaks P1 — PATIENT SAFETY
Symptom
Two related issues: (1) The alarm-OFF logic at corsano_cardiac_arrest.c:287 requires only 4 consecutive above-threshold peaks to cancel an active alarm, with no inter-peak timing validation (lines 302-330 are commented out). In scenario 23 (asystole), the alarm fires 6 times but is cancelled within seconds each time. (2) A single spurious above-threshold peak resets below_threshold_duration = 0, preventing sustained accumulation toward the alarm-ON threshold.
Affected Scenarios
23_asystole (6 alarm ON/OFF cycles, cardiac_arrest=2 for 1/60 records), 24_pea (1 alarm ON/OFF cycle, cardiac_arrest=2 for 1/60 records). Scenarios 21 and 22 never reach alarm-ON because noise peaks prevent below_threshold_duration from accumulating past 10s.
Root Cause
During cardiac arrest, noise occasionally produces spurious peaks that satisfy amplitude_difference >= threshold. These peaks: (a) reset below_threshold_duration = 0 at line 340, preventing alarm-ON, and (b) increment consecutive_above_threshold at line 287, triggering alarm-OFF after just 4 peaks. The 2-second inter-peak validation that would reject temporally scattered noise peaks is commented out (lines 302-330).
Evidence
Scenario 23 (asystole) log shows 6 complete alarm cycles: 
// Alarm cycle example from scenario 23:
Cardiac Alarm3 ON (invalid pairs)!   // alarm fires correctly
Cardiac below thresh 16.0 2.0        // still in arrest
Cardiac Alarm OFF!                   // cancelled by 4 spurious peaks

// Counter reset pattern:
Cardiac below thresh 27.8 2.0  // 27.8s accumulated
Cardiac below thresh 6.1 6.1   // RESET to 6.1s by single spurious peak

// Scenario 22 (VT→VF): 0 "below thresh" msgs -- noise peaks
// are frequent enough to prevent ANY accumulation
Fix
(1) Un-comment the 2-second inter-peak validation at lines 302-330 to reject temporally scattered noise peaks. (2) Require N consecutive above-threshold peaks (e.g., 15) before resetting counter or cancelling alarm. (3) Add amplitude gating: when green avg amp drops below ~100 (normal ~800), reject all peaks as noise. (4) Add hysteresis: alarm-OFF should require 30+ seconds of sustained normal rhythm.
BUG-3: HR Doubling by Dicrotic Notch (HR < 55 bpm) P2 — ALGORITHM
Symptom
Heart rate is consistently reported at exactly 2x the true rate for bradycardic patients. Measured/expected ratios: 2.005, 2.145, 1.967, 2.006. Algorithm reports with high confidence (quality=3).
Affected Scenarios
18_bradycardia (42→84.2 bpm, ratio 2.005), 20_bav3_complete (33→70.8 bpm, ratio 2.145), 33_hypothermia_32C (55→108.2 bpm, ratio 1.967), 34_hypothermia_28C (35→70.2 bpm, ratio 2.006)
Root Cause
Two reinforcing mechanisms exclude the true fundamental frequency, leaving only the 2nd harmonic (dicrotic notch):
  1. Bandpass filter cutoff at 0.7 Hz (42 bpm): At mmt_vital_algorithms.c:281-283, the green channel bandpass filter has lowcut=0.7. At 42 bpm, the fundamental sits at the -3dB point (50% power attenuation). At 35 bpm (0.58 Hz), it is heavily suppressed (>-10 dB). The 2nd harmonic passes unattenuated.
  2. FFT search floor at 45 bpm: At mmt_vital_algorithms.c:70, #define LOW_BPM_50 45.0 sets the FFT search window to bin 24 (45 bpm). A 42 bpm fundamental maps to bin 22.4, below the search floor. A 33 bpm fundamental maps to bin 17.6, far below. Only the doubled frequency is found.
Once locked to the doubled frequency, prev_hr > 80 raises the search floor to 50-65 bpm (lines 3626, 3695), and the jump limiter (~4 bpm/step, lines 3677-3685) prevents recovery. The algorithm permanently latches to the 2nd harmonic.

Fix (3-level approach)
  1. Lower bandpass filter cutoff (mmt_vital_algorithms.c:283): Redesign with lowcut=0.4 Hz (24 bpm) instead of 0.7 Hz (42 bpm)
  2. Lower FFT search floor (mmt_vital_algorithms.c:70): Change LOW_BPM_50 from 45.0 to 25.0 to include bradycardic fundamentals
  3. Add sub-harmonic checking: After finding the dominant FFT peak at frequency f, check for significant energy at f/2. If peak_at_f/2 / peak_at_f > 0.3, report f/2 as the true HR. Most robust approach.
NEW: SpO2 Selection Bias at Extreme R-Ratios P2 — ALGORITHM
Symptom
Scenario 06 (severe hypoxia, target SpO2=72%): measured 86.6% (+14.6 delta). Per-beat SpO2 values scatter wildly from 50 to 97 in the same window.
Root Cause
Two compounding mechanisms: (1) At R ~1.5 (SpO2 ~72%), the quality filter at mmt_vital_algorithms.c:4574 (if (R > 2) good = false) and the spo2_std > 10 filter systematically reject the most hypoxic beats (highest R values) while keeping beats where noise accidentally lowered R. This creates a positive selection bias. (2) The Proc_spo2 rolling average initializes at 100% and converges too slowly for severely low SpO2 — only 9 non-zero SpO2 values are written in 60 minutes, with early values near 99% dragging the average up to 86.6%.
Evidence
TTT1 log: R = 1.518 (expected 1.401). Per-beat scatter: Spo2s__1,92.76,87.03,64.37,70.92,92.97,70.90,67.76,61.34 — same window, values range from 61 to 93.
Fix
Implement adaptive R-ratio quality thresholds based on the running R estimate. For R > 1.2 (SpO2 < 78%), widen the acceptance window or use median instead of mean.
BUG-1: SpO2 old_mid Formula +5 Offset Bias P2 — ALGORITHM
Symptom
The old_mid SpO2 values are consistently 3-5 points higher than avg_good values.
Root Cause
The formula at mmt_vital_algorithms.c:5325 contains a hardcoded +5 offset: 
float y = 1.595 * x * x - 32.67 * x + 112.69 + 5;
This produces SpO2 = 94.8 instead of 90.5 for R = 0.726 (scenario 04).
Status
The current firmware code path uses spo2_avg_good (tuned curve) for the final SPO2_F output. However, if the analysis pipeline or any fallback path uses old_mid, the bias appears.
Fix
Remove the +5 constant from the old_mid formula, or ensure the analysis pipeline exclusively reads SPO2_F values.
BUG-6: Movement Detection Mismatch — mmt_idle_detection vs PSP Classifier P3 — SIMULATOR / FIRMWARE
Symptom
Scenario 33 (hypothermia 32°C with shivering): PSP classifier correctly classifies as REST (variance threshold raised to 20k), but mmt_idle_detection returns ACTIVE(4), causing 100% SpO2 movement rejection. SpO2=0 for all 60 minutes.
Root Cause
Two independent motion-detection systems use incompatible metrics: PSP uses per-second variance (threshold 20k), while mmt_idle_detection.c:256-270 uses sum of Euclidean sample-to-sample deltas over 100 samples. For 10 Hz shivering at 0.15g (154 counts), sum_d_acc reaches ~36,600, far above the ACTIVE threshold of 2,700 at line 175. The SpO2 filter at mmt_vital_algorithms.c:5956 then zeros SpO2 when movement_level > ALMOST_IDLE(1).
Impact
SpO2 completely suppressed during shivering despite being stationary. Also amplifies BUG-3: the low quality prevents quality-enhancement filters, making HR doubling noisier (std=2.5 vs <1.0 for pure BUG-3 scenarios).
Scenario 34 (28°C, no shivering) — Different SpO2=0 mechanism
No movement issue (IDLE). SpO2=0 because Red-IR peak alignment mismatch (time_diff 4.2-7.0s) exceeds the fixed 4.0s threshold at mmt_vital_algorithms.c:5989. At 35 bpm with 30% DC levels and noise_std=200, peak detection is too imprecise. Valid SpO2=82 readings are rejected. Fix: scale threshold with HR: threshold = max(4.0, 60.0/HR * 2.5).

Bug Interaction Diagram

After BUG-4 and BUG-5 fixes, two remaining issues prevent reliable arrest detection:
  1. BUG-4 (partially fixed): Invalid-pair alarm check now works → "Alarm3 ON (invalid pairs)!" fires in scenarios 23, 24. But VF noise (scenarios 21, 22) prevents below_threshold_duration from accumulating past 10s.
  2. BUG-5 (FIXED): Shadowed variable removed → gap-based alarm now functional.
  3. NEW: Alarm-OFF too permissive: Only 4 consecutive above-threshold peaks cancel alarm. Inter-peak timing validation is commented out (lines 302-330). During arrest, noise produces sporadic peaks satisfying the condition, causing rapid alarm ON/OFF cycling. This is now the primary remaining issue.
  4. NEW: Counter reset: A single spurious above-threshold peak resets below_threshold_duration = 0, preventing sustained accumulation toward alarm-ON.

Additional Concerns

  • False HR during arrest: All 4 scenarios report HR 72-177 bpm during arrest phases. HR comes exclusively from the internal FFT-based PR algorithm in mmt_vital_algorithms.c (PSP stub does not emit HR metric). The HRM_filtered buffer retains historical values and decays slowly.
  • No HR suppression when alarm active: The HR reporting logic at mmt_process_fxi.c:1605-1685 does not check corsano_cardiac_get_state(). Even when cardiac_arrest=2 (scenarios 23, 24), the firmware continues reporting non-zero HR. A patient in cardiac arrest should show HR=0.
  • False SpO2 during arrest: All scenarios report SpO2 95-100% during arrest — physiologically impossible.
  • LED auto-gain masks signal loss: When green amplitude drops from ~800 to ~35, LED power increases from 1% to 60%, making the signal appear "present" to the algorithm despite no actual pulse.
  • Scenario 24 (PEA) is worst case: HR reported at 80-103 bpm with 93.1% quality during pulseless rhythm. A clinician relying on the watch would see completely normal vitals during cardiac arrest.
  • Scenario 22 (VT→VF) has zero detection: VT phase produces enough above-threshold peaks that below_threshold_duration never accumulates. Zero "Cardiac below thresh" messages. HR reaches 177 bpm during VF.
